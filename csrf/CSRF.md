## 1. SOP (Same-Origin Policy)

> **브라우저의 기본 보호막** — A 사이트에서 가져온 스크립트가 B 사이트의 데이터에 함부로 접근하지 못하게 막는 정책

### 1.1 왜 SOP가 필요한가?

SOP가 없다면:
1. 사용자가 해커의 낚시 사이트(`hacker.com`)에 접속
2. 해당 사이트에 은행(`bank.com`)으로 몰래 요청을 보내는 JS가 심어져 있음
3. 브라우저가 허용 → 해커가 계좌 잔액 조회 또는 이체 실행 가능

**→ SOP는 이런 교차 출처 간 무단 데이터 접근을 원천 차단**

---

### 1.2 동일 출처(Same-Origin) 판단 기준

URL의 **3가지 요소가 모두 같아야** 동일 출처로 인정:

| 요소 | 설명 |
|------|------|
| **Protocol** | `http` vs `https` — 다르면 다른 출처 |
| **Host** | `example.com` vs `api.example.com` — 다르면 다른 출처 |
| **Port** | `:80` vs `:443` — 다르면 다른 출처 |

#### 예시: `https://v.daum.net:443` 기준 비교

| 비교 대상 URL | 결과 | 이유 |
|--------------|------|------|
| `https://v.daum.net/page1` | ✅ Same | 프로토콜, 호스트, 포트 모두 동일 |
| `http://v.daum.net` | ❌ Diff | 프로토콜 다름 (`http` vs `https`) |
| `https://news.daum.net` | ❌ Diff | 호스트 다름 (`v` vs `news`) |
| `https://v.daum.net:8080` | ❌ Diff | 포트 다름 (`443` vs `8080`) |

---

### 1.3 SOP가 제한하는 것 vs 허용하는 것

| 구분 | 대상 |
|------|------|
| **제한** (Cross-Origin Read) | `fetch`/`XHR`의 응답 데이터 읽기, `iframe` 내부 DOM 접근, 타 출처 쿠키/스토리지 읽기 |
| **허용** (Cross-Origin Write/Embed) | `<script src>` 실행, `<img>` · `<video>` · `<link>` 외부 리소스 표시, `<form>` 데이터 전송 (단, 응답값은 읽기 불가) |

---

### 1.4 SOP의 한계를 넘는 법: CORS

현대 웹은 여러 서버에서 데이터를 가져와야 하므로(구글 지도 등), SOP를 안전하게 우회하는 메커니즘이 필요 → **CORS(Cross-Origin Resource Sharing)**

서버가 응답 헤더에 아래를 포함하면 브라우저가 예외 적용:
```
Access-Control-Allow-Origin: https://my-site.com
```

> **요약**
> - **SOP**: 사이트 간 영역 침범을 막는 기본 담벼락
> - **CORS**: 이 담벼락에 합법적인 쪽문을 만드는 규칙

## 2. CSRF 개요

> **피해자의 브라우저가 피해자의 권한으로 특정 요청을 수행하게 강제하는 공격**

- HTTP의 **무상태성(Stateless)** 을 악용
- `iframe`, `img src` 등 **브라우저가 자동으로 로딩하는 태그**에 위조 요청을 심음
- 공격 페이지는 임의의 웹사이트에 설치되거나 **HTML 이메일로 전송**될 수 있음
- 피해자가 **이미 다른 탭에서 인증된 상태**라는 가정 하에 동작
  - 피해자의 세션 쿠키가 위조 요청에 자동으로 포함되어 정상 요청처럼 보임

---

## 3. 공격 방식 및 원리

### 3.1 강제 브라우징을 이용한 요청 위조

사용자 의도 없이 특정 주소로 요청을 보내게 만드는 기법:

- 피해자의 메일을 공격자에게 포워딩
- 피해자의 증권 계좌에서 주식 거래
- 피해자 계좌 → 공격자 계좌로 이체

### 3.2 이미 인증된 사용자 공격

브라우저에 남아있는 **쿠키/세션**을 활용해 로그인된 사용자의 권한을 도용

### 3.3 CSRF + XSS 복합 공격

| 공격 | 역할 |
|------|------|
| **XSS** | 브라우저에 코드를 삽입해 데이터 수집 또는 특정 동작 유발 |
| **CSRF** | 웹사이트의 기능을 타겟으로 피해자 브라우저가 공격자 대신 요청 수행 |

XSS로 CSRF 공격 코드를 실행 → **성공률과 파괴력 동시 증가**

---

## 4. 심화 및 변형 공격

### 4.1 POST를 이용한 공격

GET(URL 파라미터)뿐만 아니라 **POST 방식**에서도 위조 가능:
- 요청 URI에 데이터를 추가하거나
- 다른 서버의 요청 변환기를 이용해 **POST → GET 변환** 후 전송

### 4.2 다양한 접근 경로

API, 모바일 환경 등 웹 서비스가 제공되는 **모든 경로**가 공격 대상이 될 수 있음

### 4.3 클릭재킹 (Clickjacking)

> CSRF의 변형 — 투명한 레이어로 사용자를 속여 엉뚱한 버튼을 클릭하게 유도

**동작 방식:**
1. 타겟 페이지를 `iframe`으로 삽입
2. `opacity`(투명도)와 스타일 속성으로 숨김 처리
3. 피해자는 정상 페이지만 보이지만, 마우스 클릭은 숨겨진 버튼으로 전달

```
[정상 페이지 화면]
      ↕ (투명 iframe 겹침)
[타겟 페이지의 실제 버튼]
```

## 5. 방어법

### 5.1 입력 필터링

서버에서 요청 데이터의 유효성을 철저히 검증

### 5.2 HTTP 헤더 활용

- **`X-CSRF-Token` 헤더**: 커스텀 헤더로 요청 출처 검증 (단순 폼 요청은 커스텀 헤더 전송 불가)

### 5.3 공유 비밀 (CSRF Token)

인증된 사용자의 민감한 폼/링크에 **의사 무작위 임시 토큰**을 할당:

- 토큰은 **예측 불가능한 일회용 값**이어야 함
- 아래는 모두 **취약한** 토큰 방식:
  - 시간 기반 토큰
  - 순차적인 값 토큰
  - 사용자 이메일 해시 토큰

### 5.4 쿠키 복제

브라우저의 **SOP를 활용한 방어법**:
- SOP에 의해 한 사이트는 다른 사이트의 쿠키를 읽을 수 없음
- 요청 파라미터에 쿠키 값을 복제하여 함께 전송, 서버에서 일치 여부 검증

### 5.5 사용자 확인 요청

민감한 동작 전 **재인증(비밀번호 재입력, OTP 등)** 요구

### 5.6 안티 프레이밍 (Anti-Framing)

자신의 웹 사이트가 다른 사이트의 `<iframe>`에 삽입되는 것을 방지 → **클릭재킹 방어**

#### ① X-Frame-Options 헤더

| 값 | 의미 |
|----|------|
| `DENY` | 어떤 사이트에서도 프레임 삽입 불가 (가장 안전) |
| `SAMEORIGIN` | 동일 도메인 내에서만 허용 (가장 많이 사용) |
| `ALLOW-FROM uri` | 특정 도메인만 허용 (구형 브라우저 전용) |

```http
X-Frame-Options: SAMEORIGIN
```

#### ② Content-Security-Policy (CSP) - frame-ancestors

`X-Frame-Options`보다 더 현대적이고 세밀한 제어 가능:

```http
Content-Security-Policy: frame-ancestors 'none';    /* 전체 차단 */
Content-Security-Policy: frame-ancestors 'self';    /* 자신만 허용 */
```

## 핵심 요약

```
CSRF 성립 조건
├── 피해자가 타겟 사이트에 인증된 상태
├── 브라우저가 쿠키를 자동으로 포함하여 요청 전송
└── 서버가 요청 출처를 검증하지 않음

주요 방어 수단
├── CSRF Token (가장 일반적)
├── SameSite 쿠키 속성
├── X-Frame-Options / CSP frame-ancestors (클릭재킹 방어)
└── 재인증 (민감한 동작)
```
